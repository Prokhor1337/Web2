# Лабораторна робота 7-9: Clicker Game

## 1. Мета роботи та опис завдання
Метою лабораторної роботи 7-9 є закріплення навичок роботи з бібліотекою React та створення повноцінного односторінкового інтерактивного застосунку "Clicker Game" (гра-клікер).

Робота розроблена з використанням збирача модулів Vite та відповідає наступним вимогам:
- Створення React-компонентів.
- Робота зі станом компонента (`useState`).
- Робота з ефектами та таймерами (`useEffect`).
- Умовний рендеринг та обробка подій (`onClick`, `disabled`).
- Динамічне створення списків елементів.

---

## 2. Логіка застосунку та стан (State)
Вся логіка гри зосереджена у головному компоненті `App.jsx`. Для збереження даних, які змінюються в процесі гри та впливають на інтерфейс, використовуються хуки `useState`:
- `credits`: Баланс гравця (скільки монет у нього зараз є). Початкове значення — 0.
- `clickPower`: Сила кліку (скільки монет додається за одне натискання на головну кнопку). Початкове значення — 1.
- `autoIncome`: Пасивний дохід (скільки монет автоматично додається щосекунди). Початкове значення — 0.

Так як у React зміна стану викликає автоматичне оновлення екрану (ререндер), будь-яка зміна цих змінних миттєво відображається у блоці статистики.

---

## 3. Обробка подій та ігровий цикл (Effects)
### Головний клік
Для обробки ручного натискання на кнопку "Клік!" створено функцію `handleMainClick`. Вона використовує функціональний підхід для оновлення стану: `setCredits(prev => prev + clickPower)`. Це гарантує, що до попереднього актуального балансу буде додано поточну силу кліку.

### Автоматичний (пасивний) дохід
Для реалізації пасивного доходу використовується хук життєвого циклу `useEffect`:
- Він слідкує за зміною змінної `[autoIncome]`.
- Якщо дохід більший за 0, запускається інтервал `setInterval`, який кожну 1000 мс (1 секунду) додає `autoIncome` до балансу `credits`.
- **Очищення ефекту**: всередині ефекту обов'язково повертається функція `return () => clearInterval(timer)`. Це критично важливий етап в React, який запобігає витоку пам'яті (memory leak) та не дозволяє таймерам накладатись один на одного при перемальовуванні компонента.

## 4. Динамічний рендеринг списку покращень
Замість того, щоб жорстко прописувати кожну кнопку покращення в HTML, усі доступні апгрейди зберігаються у масиві об'єктів `upgrades`.
Кожен об'єкт містить `id`, назву, тип (`click` для ручного або `auto` для пасивного), розмір покращення та вартість.

У JSX-розмітці використовується метод масиву `map()`:
```javascript
{upgrades.map(u => (
  <button key={u.id} disabled={credits < u.cost} onClick={() => buyUpgrade(u)}>
    ...
  </button>
))}
```
- **Унікальний ключ**: Кожному елементу списку передається обов'язковий атрибут `key={u.id}`, щоб React міг оптимізувати рендеринг.
- **Умовне блокування**: Атрибут `disabled={credits < u.cost}` перевіряє, чи вистачає у гравця кредитів. Якщо баланс менший за ціну, кнопка стає неактивною і змінює свій стиль, не дозволяючи зробити клік.
- **Покупка**: При натисканні викликається функція `buyUpgrade(u)`. Вона віднімає вартість покращення від балансу, а потім перевіряє тип (`u.type`). Якщо це `click`, вона підвищує `clickPower`, а якщо `auto` — підвищує `autoIncome`.

## 5. UI та Стилізація
Дизайн побудований за допомогою звичайного CSS (`App.css`).
Інтерфейс логічно розбитий на 3 частини:
1. **Статистика (`.stats`)**: Має темний контрастний фон, який ідеально читається як на світлій, так і на темній темі Vite.
2. **Головна кнопка (`.main-btn`)**: Велика, яскрава помаранчева кнопка із ефектом анімації при натисканні (зменшується завдяки `transform: scale()`).
3. **Меню покращень (`.upgrades`)**: Організовано у вигляді вертикального списку (`flex-direction: column`). Кнопки змінюють прозорість при недостатній кількості кредитів, даючи гравцеві візуальний зворотний зв'язок.
